# 实现计划书（服务 / 仓库维度）

> 说明
>
> - 本文档基于现有设计文档整理，聚焦“如何落地”，不重复讨论产品愿景。
> - 按仓库 / 服务拆分任务，便于逐项分配、排期、验收。
> - 用户补充的新开发计划已纳入：
>   - 前端支持中英双语
>   - 文档支持中英双语
>   - 开发内容注释统一使用中文（函数注释、文件注释）
> - 本计划不参考 `roadmap.md`，仅依据当前设计文档与本次需求整理。
> - 文档中的“已实现 / 未实现”状态不作为计划依据；计划以“要做什么、怎么做、怎么验收”为准。

---

## 1. 计划目标

### 1.1 总目标

在不破坏现有 API / 数据库 / 服务间通信设计的前提下，完成一套可持续交付的实现计划，覆盖：

1. 平台各服务按设计文档完成核心能力落地。
2. Web 前端具备中英双语能力，所有核心用户界面可切换语言。
3. 产品与技术文档具备中英双语版本管理能力。
4. 代码注释规范统一，新增和重构代码中需要补齐中文函数注释与文件注释。
5. 各服务之间的接口、数据模型、文档、测试和验收标准保持一致。

### 1.2 成功标准

当本计划执行完成后，应达到以下结果：

- 用户可以在前端切换中文 / 英文，页面文案、导航、表单提示、错误提示、空状态、确认文案等均能跟随语言变化。
- 文档站点或仓库文档中，核心设计、产品说明、使用说明至少具备中英可读版本。
- 代码仓库中新增或重点修改的函数、关键文件具备中文注释，且注释表达清晰、可维护。
- API、数据库、服务通信与设计文档一致，无明显实现偏差。
- 测试可覆盖关键主流程、国际化切换、文档产出与注释规范检查。

---

## 2. 适用范围与边界

### 2.1 适用范围

本文档覆盖以下仓库 / 领域：

- Web 前端仓库
- Gateway / API 网关相关仓库
- IAM / Project / Spider / Execution / Scheduler / Node / Datasource / Monitor 等后端服务仓库
- 文档仓库或仓库内 `docs/` 目录
- 测试与质量保障脚本、约定、CI 配置

### 2.2 不纳入本次计划的内容

以下内容不作为本计划的优先目标：

- `roadmap.md` 的内容整理或重写
- 与当前设计文档无直接关系的全新业务功能
- 大规模重构但不影响本次目标的历史遗留代码清理
- UI 视觉风格重设计（除非为支持国际化而必要）

### 2.3 原则

- 先保证主链路可用，再做体验增强。
- 先建立统一规范，再批量扩展到各模块。
- 先改底层可复用能力，再改业务页面和文档。
- 国际化、文档双语、注释规范三件事必须并行规划，但允许分阶段实施。

## 3. 总体实施策略

### 3.1 推荐策略

建议采用“基础设施先行 + 各服务并行落地 + 验收逐层推进”的策略：

1. 先统一国际化规范、文档规范、注释规范。
2. 再在前端建立语言系统、资源文件、切换入口与 fallback 机制。
3. 同步推进各后端服务的文档化、错误响应说明、注释补齐。
4. 最后做跨仓库验收、术语统一、翻译校对和回归测试。

### 3.2 为什么采用这个策略

- 可以避免前端、后端、文档各自定义语言和术语，导致后期返工。
- 可以让翻译资源、术语表、注释模板、文档模板复用。
- 可以确保后续新增功能时直接继承双语能力，而不是每次重新补。

## 4. 仓库 / 服务级实现计划

### 4.1 Web 前端仓库

#### 4.1.1 目标

让 Web 前端具备完整的中英双语能力，并保证：

- 页面语言可切换
- 切换后当前页面尽量不刷新或少刷新
- 路由、菜单、表单、弹窗、空状态、错误提示、通知等全部支持国际化
- 可持续扩展新的语言资源

#### 4.1.2 需要实现的内容

##### A. 国际化基础框架

1. 引入并确认前端国际化方案。
2. 建立语言资源目录结构。
3. 建立中文为默认语言、英文为可切换语言的机制。
4. 建立语言切换入口：
   - 顶部导航
   - 设置页
   - 登录后用户菜单
5. 建立语言持久化策略：
   - localStorage / cookie / URL 参数 三选一或组合
   - 优先级明确
6. 建立语言 fallback 规则：
   - 缺失英文时回退中文，或缺失中文时回退英文
   - 禁止页面出现未翻译 key

##### B. 页面文案国际化

按页面逐步替换静态文本为 key：

1. 登录 / 注册页
2. 项目列表 / 项目创建页
3. Spider 列表 / 版本管理页
4. Execution 创建 / 详情 / 日志页
5. Schedule 列表 / 创建 / 编辑页
6. Node 列表 / 详情 / 心跳状态页
7. Datasource 管理页
8. Monitor 仪表盘 / 告警规则 / 告警事件页
9. 通用组件：
   - 搜索栏
   - 表格列名
   - 分页
   - 弹窗
   - 表单校验
   - Toast / Notification
   - Loading / Empty / Error 状态

##### C. 路由与导航国际化

1. 菜单名称国际化。
2. 面包屑国际化。
3. 页面标题国际化。
4. 404 / 403 / 500 等错误页国际化。
5. 若存在 URL 中的固定中文展示，应统一改为语言资源渲染。

##### D. 表单与校验国际化

1. 输入框 label / placeholder / helper text 国际化。
2. 后端错误信息映射为前端可读语言。
3. 表单校验提示统一通过语言资源输出。
4. 针对 create / update / delete 等操作确认弹窗进行国际化。

##### E. 后端错误与前端映射

1. 统一错误码与错误消息映射表。
2. 对常见错误做前端可翻译处理：
   - 401 未登录
   - 403 无权限
   - 404 资源不存在
   - 409 冲突
   - 500 服务异常
3. 保留服务端原始 error 文本作为调试信息，不直接暴露为唯一展示文案。

##### F. 前端测试

1. 语言切换测试：
   - 默认语言
   - 切换后刷新保持
   - 切换后路由不丢失
2. 关键页面快照测试：
   - 中文版页面
   - 英文版页面
3. 通用组件测试：
   - 文案 key 缺失时 fallback
   - 长文本换行
   - 按钮宽度适配
4. 交互测试：
   - 提交成功
   - 提交失败
   - 错误提示翻译

##### G. 共享组件（状态：已完成 2026-05-05）

基于 Element Plus 完成了以下共享组件开发和迁移，涵盖 Section A（国际化基础框架）、B（页面文案）、C（路由导航）、D（表单校验）、E（错误映射）、F（前端测试）：

**现有组件迁移为 Element Plus：**
1. AppLayout → `el-container` + `el-header` + `el-menu`（horizontal 模式，路由联动高亮，activeIndex 自动匹配）
2. AppLanguageSwitcher → `el-select`（小尺寸，中文/English 标签，响应式追随 locale store）
3. AppEmptyState → `el-empty`（description 走 `localeStore.t()` 读取 i18n key）
4. AppLoadingState → `el-skeleton` + `v-loading` 指令（skeleton / text 双模式，支持 `rows` 配置）

**新增共享组件：**
1. AppTable — 通用表格（`el-table` + `el-pagination`），泛型 typed，支持排序/分页/加载/空状态，列头走 i18n key
2. AppForm — 通用表单（`el-form`），支持 6 种字段类型（input / textarea / number / select / switch / date），表单校验和按钮文案走 i18n
3. AppConfirmDialog — 操作确认弹窗（`ElMessageBox` 封装），含 `confirmAction` 通用函数和 `confirmDelete` 便捷函数
4. AppNotification — Toast 通知工具（`ElMessage`），导出 `notifySuccess` / `notifyError` / `notifyWarning` / `notifyInfo`
5. AppErrorState — 错误状态展示 + 重试按钮（`el-result`），message/retry 文案走 i18n
6. AppBreadcrumb — 面包屑导航（`el-breadcrumb`），items 支持 to/labelKey

**相关增强：**
- `el-config-provider` 语言包随 `localeStore` 响应式切换（App.vue 提供 `elLocale`）
- `api/client.ts` 增强为 `ApiError` 类，HTTP 状态码自动映射为 i18n 错误码
- `messages.ts` 补齐 50+ 个 i18n key：nodes / schedules / errors / actions / notFound / gateway
- 全部 9 个视图页面（Login / Projects / Spiders / Executions / ExecutionDetail / Schedules / Nodes / Datasources / Monitor）迁移到 Element Plus 组件 + i18n key
- 所有 22 个测试通过，Vite 构建成功

#### 4.1.3 前端交付物

- 国际化基础模块
- 中英资源文件
- 语言切换入口
- 页面文案替换清单
- 前端国际化测试用例

#### 4.1.4 风险点

- 某些页面中存在硬编码中文，需要逐项排查。
- 表格列名、状态枚举、错误提示可能分散在多个组件中，容易遗漏。
- 英文文案长度更长，布局可能需要微调。

### 4.2 Gateway 仓库

#### 4.2.1 目标

保证网关层对前端双语不构成阻塞，同时为前端提供稳定、可国际化消费的错误响应。

#### 4.2.2 需要实现的内容

1. 统一错误响应结构，确保前端可映射语言。
2. 保持请求链路中的 request-id、access log 可追踪。
3. 对 JWT 校验失败、内部令牌校验失败、限流等场景输出稳定错误码。
4. 若网关层存在文案，补充中文注释和必要说明。
5. 检查 API 版本路由对前端切换语言无副作用。

#### 4.2.3 验收点

- 网关错误响应可被前端统一处理。
- 语言切换不影响鉴权、限流和路由转发。
- 网关无新增硬编码文案。

### 4.11 文档仓库 / docs 目录

#### 4.11.1 目标

建立中文与英文并存的文档体系，使设计、产品、使用说明能够被中文和英文读者理解。

#### 4.11.2 需要实现的内容

##### A. 文档结构规划

建议采用以下结构：

- `docs/zh-CN/`：中文主文档
- `docs/en-US/`：英文主文档
- `docs/shared/`：术语表、图表源文件、共享模板
- `docs/product/`：产品文档
- `docs/design/`：设计文档
- `docs/architecture/`：架构文档
- `docs/how-to/`：使用说明与操作手册

##### B. 双语文档映射

至少保证以下文档存在中英对应版本或双语主体：

1. 产品概览
2. MVP / 阶段说明
3. API 设计
4. 数据库设计
5. 服务间通信设计
6. 前端设计
7. 操作文档 / 部署文档 / 本地开发文档
8. 变更记录 / 发布说明

##### C. 文档写作规范

1. 中文文档为主文档时，英文文档需保持结构一致。
2. 术语表统一管理，避免同一概念多种翻译。
3. 标题层级、列表结构、表格字段保持一致，便于双向维护。
4. 设计图、表格、接口示例尽量双语说明。
5. 所有新写入文档，至少需要具备：
   - 中文说明
   - 英文摘要或英文全文

##### D. 维护方式

1. 新增文档时同步创建双语版本。
2. 修改核心设计文档时，优先同步更新双语对应文件。
3. 若短期无法完整翻译，可先提供英文摘要，但必须标记后续补全任务。

#### 4.11.3 验收点

- 核心文档可以按语言目录独立阅读。
- 术语不会在中英文版本中漂移。
- 文档结构与内容能对应追踪。

---

### 4.12 代码注释规范与执行要求

#### 4.12.1 目标

让新增和重点修改代码具备可读、可维护的中文注释。

#### 4.12.2 需要实现的内容

##### A. 函数注释

以下类型函数应补中文注释：

1. 业务核心函数
2. 数据转换函数
3. 状态计算函数
4. 外部调用封装函数
5. 复杂条件分支函数
6. 重试 / 调度 / 消息投递 / 同步逻辑函数

注释应至少说明：

- 这个函数做什么
- 输入输出含义
- 关键副作用或边界条件
- 如有必要，说明为什么这样实现

##### B. 文件注释

以下类型文件应补中文文件注释：

1. 单文件职责明显的业务模块
2. 与外部系统交互的适配层
3. 复杂状态机 / 队列 / 调度 / 告警处理模块
4. 公共工具类、公共配置、公共常量文件

文件注释应说明：

- 该文件负责什么
- 与谁交互
- 不负责什么

##### C. 注释边界

1. 不对显而易见的一行代码做无意义注释。
2. 不写重复代码含义的“翻译式注释”。
3. 优先给复杂逻辑、公共边界、容易误解处补充说明。
4. 注释必须与实现同步更新，避免过期。

#### 4.12.3 验收点

- 新增核心函数均可读。
- 新增复杂文件均有中文说明。
- 注释覆盖“为什么这么做”，而不是只重复“做了什么”。

## 5. 统一语言体系设计

### 5.1 前端语言策略

建议采用以下语言体系：

- 默认语言：中文 `zh-CN`
- 第二语言：英文 `en-US`
- 切换策略：用户可手动切换，浏览器语言可作为首次默认值参考
- 持久化策略：记住用户上次选择

### 5.2 术语表策略

建议建立统一术语表，至少覆盖：

- Project / 项目
- Spider / 爬虫
- Spider Version / 爬虫版本
- Execution / 执行
- Schedule / 调度任务
- Node / 节点
- Datasource / 数据源
- Alert Rule / 告警规则
- Alert Event / 告警事件
- Registry Auth Ref / 镜像仓库凭据引用

### 5.3 文案风格要求

中文：

- 简洁、明确、避免口语化歧义

英文：

- 简洁、规范、避免直译中文语序

### 5.4 翻译边界

- 状态值、枚举值、错误码、字段名应保持稳定，展示层翻译。
- 不应直接翻译数据库字段名或 API 参数名。
- 前端文案与文档说明必须使用同一术语表。

## 9. 风险与对策

### 9.1 风险：翻译资源膨胀

**表现**：页面多、文案多、key 管理复杂。  
**对策**：

- 建立命名规范
- 按服务拆分命名空间
- 建立共享通用 key
- 避免重复 key

### 9.2 风险：术语不一致

**表现**：同一功能在不同页面和文档中翻译不统一。  
**对策**：

- 统一术语表
- 统一审校人
- 先冻结核心术语再批量翻译

### 9.3 风险：文案过长导致布局异常

**表现**：英文按钮、标签、说明文字超出布局。  
**对策**：

- 提前做英文长度回归
- 使用自适应布局
- 对关键按钮和表格列预留扩展空间

### 9.4 风险：注释质量不稳定

**表现**：注释过多、过少或过期。  
**对策**：

- 只要求复杂逻辑和核心模块补注释
- 建立注释模板
- 代码变更与注释同步审查

### 9.5 风险：文档双语同步成本高

**表现**：中文文档更新后，英文版本遗漏。  
**对策**：

- 双语文档成对提交
- 核心文档优先双写
- 通过模板和检查项降低遗漏

## 10. 最终验收标准

以下条件全部满足时，可视为本实现计划完成：

1. 前端核心功能页面支持中英切换。
2. 文档核心内容至少具备中英可读版本。
3. 新增和重点修改代码中的函数注释、文件注释以中文表达，且可维护。
4. 关键 API、数据库字段、通信链路与设计文档一致。
5. 主流程测试与国际化回归测试通过。
6. 核心术语在页面和文档中保持统一。

## 12. 按仓库拆分的可分派任务清单

> 说明：以下清单已经按照“可分派、可排期、可验收”的标准拆解。每个仓库内的任务可继续拆成多个工单，建议按“基础能力 → 页面/接口 → 测试 → 文档/注释 → 验收”的顺序执行。

### 12.1 Web 前端仓库任务清单

#### 12.1.1 基础国际化底座

- 选择并落地前端 i18n 方案，统一语言包加载方式。
- 约定语言资源目录、命名空间和 key 命名规范。
- 实现语言持久化读取与写入，优先级明确为：用户手动选择 > 本地缓存 > 浏览器语言 > 默认中文。
- 增加全局语言切换入口，并保证切换后当前页面状态尽量保留。
- 建立通用翻译工具方法，支持占位符、复数、动态拼接和 fallback。

#### 12.1.2 通用组件国际化

- 将按钮、标签、弹窗、Toast、表单校验、分页、空状态、Loading、错误页等公共组件改为读取语言资源。
- 对表格列头、状态标签、时间格式、数量单位等公共展示项做统一翻译封装。
- 排查硬编码中文字符串，替换为语言 key。
- 为长英文文案预留样式适配，避免按钮和标题溢出。

#### 12.1.3 页面级国际化

- 登录页、注册页、找回密码页如存在，则全部替换为双语文案。
- 项目、Spider、Execution、Schedule、Node、Datasource、Monitor 等核心页面按模块替换静态文本。
- 路由标题、面包屑、菜单名称、权限提示、空状态提示统一国际化。
- 对页面中枚举状态统一建立“状态码 → 展示文案”映射。

#### 12.1.4 错误与异常处理

- 将 401 / 403 / 404 / 409 / 500 / 502 等常见错误映射为前端可翻译文案。
- 统一网络错误、超时错误、权限错误、校验错误的展示样式与文案策略。
- 保留后端原始错误字段用于调试，但不直接作为唯一展示文本。

#### 12.1.5 前端测试任务

- 增加 i18n 初始化与 fallback 单测。
- 增加中英切换后页面状态保持测试。
- 增加核心页面中英快照测试。
- 增加长文案、窄屏、分页、表格列溢出等适配测试。
- 增加未翻译 key 检测或构建期检查。

#### 12.1.6 前端验收标准

- 中英文切换入口可用且持久化有效。
- 核心页面无硬编码可见中文或英文残留。
- 关键交互链路在两种语言下均可正常完成。
- 翻译 key 缺失时有明确 fallback，不出现空白文案。

### 12.2 Gateway 仓库任务清单

#### 12.2.1 统一错误响应

- 固定网关错误返回结构，保证前端可统一映射。
- 为鉴权失败、内部令牌失败、限流失败、版本不匹配等场景定义稳定错误语义。

#### 12.2.2 路由与鉴权兼容性

- 验证语言切换不会影响路由转发。
- 验证 JWT 校验、内部 token 校验、限流策略在多语言下行为一致。
- 若网关响应头或日志字段需要说明，补齐中文文件注释。

#### 12.2.3 Gateway 验收标准

- 网关返回错误结构稳定可预测。
- 前端能统一处理网关错误，不依赖中文/英文原文。
- 网关层不引入新的界面文案耦合。

### 12.11 文档仓库 / docs 目录任务清单

#### 12.11.1 文档双语结构建设

- 建立中文主文档目录与英文对应目录。
- 为产品文档、设计文档、架构文档、部署文档、开发文档建立统一模板。
- 建立术语表，固定核心术语翻译，避免版本漂移。

#### 12.11.2 核心文档双语化

- 为 API 设计、数据库设计、服务通信设计、前端设计、MVP 说明、使用说明等文档补齐中英版本。
- 对图片、表格、接口示例、字段解释进行双语校对。
- 对已有文档按“先结构一致，再逐段翻译”的方式逐步补全。

#### 12.11.3 文档维护流程

- 新文档创建时同步生成中英文版本。
- 修改核心文档时同步更新对应语言版本。
- 若短期无法完整翻译，则至少提供英文摘要并显式标记待补全内容。

#### 12.11.4 文档验收标准

- 核心文档可按语言独立阅读。
- 中英文版本结构一致，术语一致，示例一致。
- 不出现一份文档更新、另一份文档长期遗漏的情况。

### 12.12 代码注释规范与质量保障任务清单

#### 12.12.1 注释规范落地

- 制定函数注释模板，统一中文描述方式。
- 制定文件注释模板，说明模块职责、依赖和边界。
- 明确哪些函数必须注释：核心流程、状态转换、数据转换、外部调用、重试调度、错误处理。

#### 12.12.2 注释质量检查

- 在代码评审中加入注释检查项。
- 为新增和重点修改文件建立人工检查清单。
- 避免“重复代码含义”的无效注释，要求注释说明“为什么这样做”。

#### 12.12.3 注释验收标准

- 核心模块具备中文文件注释。
- 复杂函数具备中文函数注释。
- 注释与实现同步，不出现明显过期说明。

### 12.13 跨仓库公共任务清单

#### 12.13.1 统一术语表与翻译词库

- 建立公共术语表并冻结核心术语。
- 统一 Project / Spider / Execution / Schedule / Node / Datasource / Alert 等术语翻译。
- 约定错误码展示文案和状态枚举翻译规则。

#### 12.13.2 统一测试与验收

- 建立前端中英切换冒烟测试。
- 建立核心页面快照回归测试。
- 建立文档双语一致性检查清单。
- 建立注释检查清单和代码评审要求。

#### 12.13.3 公共任务验收标准

- 全局术语一致。
- 全局语言切换行为一致。
- 全局文档维护方式一致。
- 全局注释规范执行一致。
