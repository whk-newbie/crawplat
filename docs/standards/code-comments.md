# 代码中文注释规范

> 适用范围：Go 后端服务、gateway、agent、调度、队列、状态机、关键前端工具函数。

## 总原则

注释解释“为什么”和“边界”，不要机械复述代码。

推荐写注释的场景：

- 包或文件职责；
- 中间件职责；
- 状态机流转；
- 队列 claim / ack / retry 语义；
- 调度物化与防重复逻辑；
- 认证、鉴权、安全边界；
- 外部系统连接、探测、预览；
- 容易误改的兼容逻辑。

不推荐写注释的场景：

- `i++`、字段赋值等显而易见逻辑；
- 与代码完全重复的翻译式注释；
- 已过期的历史解释；
- 没有维护价值的情绪化说明。

## 文件注释

关键文件顶部可写一段职责说明：

```go
// 本文件负责 gateway 的 API 版本路由注册与版本前缀重写。
// 新增公共 API 时应在这里保持路由和上游服务边界一致。
```

文件注释应说明：

1. 文件负责什么；
2. 与哪些模块交互；
3. 修改时要注意的边界。

## 函数注释

导出函数按 Go 习惯以函数名开头；非导出关键函数也建议写中文职责注释。

```go
// NewRouter 构建 gateway 的统一入口路由，集中挂载观测、限流、鉴权与版本路由能力。
func NewRouter() *gin.Engine {
    // ...
}
```

函数注释应优先覆盖：

- 输入代表什么；
- 输出代表什么；
- 是否有副作用；
- 错误或失败语义；
- 并发、安全、幂等要求。

## 状态机注释

状态机相关注释必须说明允许的状态转换：

```text
pending -> running -> succeeded
pending -> running -> failed
failed -> pending（retry materialize 后的新执行）
```

不要只写“更新状态”。

## 中间件注释

中间件注释至少说明：

- 它保护或处理什么边界；
- 失败时返回什么；
- 是否会修改请求或响应；
- 是否会向上下文写入值。

## 测试注释

测试通常不需要大量注释，测试名应表达行为。只有测试夹具复杂时才补充说明。

## 注释语言

- 平台代码中的解释性注释使用中文。
- API 字段、环境变量、状态枚举保持英文原文。
- 用户可见错误消息不在注释中翻译为另一套术语。

## 抽检标准

合并前抽检：

- [ ] 新增关键文件是否有职责说明。
- [ ] 新增复杂函数是否解释了业务边界。
- [ ] 是否存在明显无意义注释。
- [ ] 注释是否与当前代码一致。
- [ ] 术语是否符合 `docs/standards/platform-terminology.md`。
