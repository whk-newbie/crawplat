# CI Lint 检查约定

> 本文档定义代码注释、文档一致性和术语使用等自动化检查的约定。可作为 CI 规则配置和代码评审的参考基准。

## 1. 中文注释检查

### 1.1 函数注释要求

以下类型的函数必须包含中文注释：

- 业务核心函数（创建、查询、更新、删除等主流程）
- 数据转换函数（DTO 转换、序列化、反序列化）
- 状态计算函数（状态机转换、状态判定）
- 外部调用封装函数（HTTP 调用、数据库操作封装）
- 复杂条件分支函数（超过 3 个分支）
- 重试 / 调度 / 消息投递 / 同步逻辑函数

注释内容应说明：

- 函数的功能
- 输入输出的含义
- 关键的副作用或边界条件
- 非显而易见的实现原因

### 1.2 文件注释要求

以下类型的文件必须包含中文文件注释：

- 单文件职责明确的业务模块
- 与外部系统交互的适配层
- 复杂状态机 / 队列 / 调度 / 告警处理模块
- 公共工具类、公共配置、公共常量文件

文件注释应说明：

- 该文件负责什么
- 与谁交互
- 不负责什么

### 1.3 注释质量规则

- 不对显而易见的单行代码添加无意义注释
- 不写仅重复代码含义的「翻译式注释」
- 优先为复杂逻辑、公共边界、容易误解之处补充说明
- 注释必须与实现同步更新，避免过期

### 1.4 检查方式

推荐在 CI 中通过以下方式进行检查：

- 使用 `grep` 或自定义脚本检查新增 Go 文件中是否包含中文注释模式 `// .*[\x{4e00}-\x{9fff}]`
- 对关键模块目录（`apps/*/internal/service/`、`apps/*/internal/model/`）进行抽样检查
- 在 PR 评审中人工确认注释质量

## 2. 双语文档一致性检查

### 2.1 文件成对检查

对 `docs/` 目录中的双语文档，检查以下规则：

- 每个 `.zh-CN.md` 文件应有对应的 `.en-US.md` 文件（核心文档适用）
- 每个 `.en-US.md` 文件应有对应的 `.zh-CN.md` 文件（核心文档适用）
- 检查脚本示例：

```bash
# 检查 product 目录下是否存在对应的英文文档
for f in docs/product/*.zh-CN.md; do
  en="${f/.zh-CN.md/.en-US.md}"
  [ -f "$en" ] || echo "Missing: $en"
done
```

### 2.2 标题层级检查

- 双语文档必须保持相同的标题层级（`#`、`##`、`###` 等）
- 标题数量应一致
- 检查脚本示例：

```bash
# 比较两个文档的标题数量
zh_headings=$(grep -c '^#' "doc.zh-CN.md")
en_headings=$(grep -c '^#' "doc.en-US.md")
[ "$zh_headings" -eq "$en_headings" ] || echo "Heading count mismatch"
```

### 2.3 文档更新检查

- 修改核心双语文档时，PR 中必须同时包含两种语言版本的变更
- 若短期无法完整翻译，需在文档中显式标记：
  - `> Translation pending. Source document: xxx.zh-CN.md.`
  - `> 待翻译。源文档：xxx.en-US.md。`
- 不允许使用模糊占位如 `TODO`、`TBD`

## 3. 术语一致性检查

### 3.1 术语表引用

核心术语必须以 `docs/standards/platform-terminology.md` 为权威来源。

关键术语清单：

| 中文 | 英文 |
|------|------|
| 项目 | Project |
| 爬虫 | Spider |
| 爬虫版本 | Spider Version |
| 执行 | Execution |
| 调度 | Schedule |
| 节点 | Node |
| 数据源 | Datasource |
| 告警规则 | Alert Rule |
| 告警事件 | Alert Event |
| 镜像仓库凭据引用 | Registry Auth Ref |

### 3.2 术语检查规则

- 同一概念在文档和代码注释中必须使用一致的翻译
- API 字段名、状态值、错误码不翻译
- 前端 i18n key 的文案必须与术语表一致

## 4. PR 检查清单

每个 PR 建议包含以下自查：

- [ ] 新增或修改的核心函数有中文注释
- [ ] 新增或修改的关键文件有中文文件说明
- [ ] 双语文档修改成对提交
- [ ] 术语使用与术语表一致
- [ ] 无不明确的 `TODO` / `TBD` 占位
- [ ] 注释未与代码实现脱节
